# 线程池详解

> 适用读者：C++ 基础薄弱、没接触过多线程和线程池的初学者。

---

## 一、前置知识：什么是线程？为什么需要池？

### 线程是什么？

想象一个餐厅：

| 概念 | 类比 | 说明 |
|------|------|------|
| **程序** | 餐厅的菜单 | 写好了菜谱，但还没开始做 |
| **进程** | 一家正在营业的餐厅 | 程序跑起来了，有厨房、有服务员 |
| **线程** | 一个厨师 | 真正干活的人，一个进程里可以有多个线程同时干活 |

```cpp
// main.cpp 的第 63 行
BS::thread_pool pool(6);    // 创建了 6 个厨师！
```

**一个厨师 = 一个线程**。6 个线程 = 6 个活可以同时干。

### 为什么需要"池"？

如果你每次需要干活就去外面现招一个厨师，干完再把厨师辞退：

```
招厨师（很慢）→ 炒一个菜（很快）→ 辞退厨师（浪费）
再招厨师（又很慢）→ 炒一个菜（很快）→ 辞退厨师（又浪费）
...
```

这样效率极低，招人比干活还久。

**线程池的思路：** 提前招好 6 个厨师，让他们在厨房等着。有活来了就扔给他们干，干完不辞退，继续等下一个活。

```
池里固定6个厨师（开一次，用一辈子）
    厨师1 ———————————————— 做菜 ———————— 等着 ———————— 做菜 ————————
    厨师2 ———————— 做菜 ———————— 等着 ————————————————————————————
    厨师3 ————————————————————— 做菜 ———————— 等着 ——————
    ...
```

## 三、第一套：spdlog 的异步日志线程池

### 代码（只有一行）

```cpp
// mas_log.cpp 第 40 行
spdlog::init_thread_pool(65536, 1);
```

### 两个参数是什么意思？

```cpp
spdlog::init_thread_pool(65536, 1);
//                        ↑      ↑
//                     缓冲区   后台线程数
```

| 参数 | 值 | 大白话 |
|------|-----|--------|
| `65536` | 缓冲区大小（字节） | 一块 64KB 的"便签墙"，日志消息先贴在上面 |
| `1` | 后台线程数 | 派 **1 个专人** 在后台负责把便签上的内容抄到日记本（磁盘文件）上 |

### 为什么要异步？

假设你写代码：

```cpp
MAS_LOG_INFO("某件事情发生了");
MAS_LOG_INFO("另一件事情发生了");
MAS_LOG_INFO("再一件事情发生了");
```

如果不用异步（同步模式），每次调用 `MAS_LOG_INFO` 时：

```
主线程：记录日志 → 打开文件 → 写入磁盘 → 关闭文件 → 继续干活
         ↑________________磁盘I/O，非常慢________________↑
```

主线程被卡住了，程序会卡顿。

**异步模式：**

```
主线程： 把消息扔进缓冲区 → 继续干活（根本不碰磁盘）
               ↓
后台线程：（慢慢写磁盘，不卡主线程）
```

```
主线程  ──→  缓冲区（内存，快）  ──→  后台线程  ──→  磁盘文件（慢）
  ↑                                               ↑
  只管扔进去                                    专人负责
```

### 一条日志的一生

```
MAS_LOG_INFO("hello") 被调用
        │
        ▼
消息被塞进缓冲区（64KB的内存区域）
        │
        ▼                           ┌─ 缓冲区快满了？
后台线程被唤醒                       │
        │                           ├─ 到3秒了？
        ▼                           │
从缓冲区取出消息                      ├─ 遇到warn及以上级别的日志？
        │                           │
        ▼                           └─ 以上任意一个触发
写到日志文件 "logs/mas_vision_xxx.log"
```

### 第 54-56 行的异步 logger 和这个线程池的关系

```cpp
// mas_log.cpp 第 54-56 行
logger_ = std::make_shared<spdlog::async_logger>(
    "mas_logger", sinks.begin(), sinks.end(),
    spdlog::thread_pool(),        // ← 用第 40 行创建的那个线程池！
    spdlog::async_overflow_policy::overrun_oldest);
```

```cpp
spdlog::thread_pool()     // 获取第 40 行创建的那个线程池
```

就像一个完整的快递站：

```
spdlog::init_thread_pool(65536, 1)   →   搭建了仓库 + 招了一个快递员
spdlog::thread_pool()                →   拿到那个快递站
spdlog::async_logger(...)            →   告诉logger："用这个快递站发货"
```

### `overrun_oldest` 是什么？

如果日志太多了，后台线程来不及写，缓冲区满了怎么办？

```cpp
spdlog::async_overflow_policy::overrun_oldest
```

| 策略 | 行为 |
|------|------|
| `overrun_oldest` | 扔掉**最旧的**消息，保留新的 |
| `block` | 主线程**等着**，等缓冲区空出来再继续 |

这个项目选 `overrun_oldest`：宁可丢几条旧日志，也别卡主线程。

## 四、第二套：BS::thread_pool —— 项目的任务线程池

### 代码

```cpp
// main.cpp 第 63 行
BS::thread_pool pool(6);
```

这一行干了什么？

```
创建了 6 个线程（6个厨师）
        │
        ├── 线程0 ── 等着接活
        ├── 线程1 ── 等着接活
        ├── 线程2 ── 等着接活
        ├── 线程3 ── 等着接活
        ├── 线程4 ── 等着接活
        └── 线程5 ── 等着接活
```

### 怎么向池里交任务？

`submit_task()`：把活交给池子，池子随便分配一个空闲线程去干。

```cpp
// main.cpp 第 86-87 行
auto camera_future = pool.submit_task(
    [camera_buffer_size, camera_buffer]() {
        threads::camera_thread_func("config/hikcamera.yaml",
                                     camera_buffer_size,
                                     camera_buffer);
    });
```

一步步拆开：

| 代码 | 含义 |
|------|------|
| `pool.submit_task(...)` | 把活交给池子 |
| `[...]() { ... }` | 这是一个 **lambda 表达式**（匿名函数），把"要干的活"打包成一个包裹 |
| `camera_buffer_size, camera_buffer` | 捕获列表，把外部变量**拷贝**进包裹里，线程才能用到 |
| `threads::camera_thread_func(...)` | 实际要执行的函数 |
| `auto camera_future = ` | 拿到一个"收据"（future），到时候靠它知道活干完了没 |

### future 是什么？

你去干洗店洗衣服：

```
1. 你把衣服给店员  →  pool.submit_task(任务)
2. 店员给你一张收据  →  future
3. 你去干别的事       →  main 函数继续执行
4. 凭收据取衣服       →  future.get()
```

```cpp
// main.cpp 第 130-133 行
if (camera_future.valid())      // 收据有效吗？
{
    camera_future.get();         // 凭收据取结果（阻塞等待线程干完活）
}
```

### 项目中往池子里塞了哪些任务？

```cpp
// main.cpp 中一共往池子里扔了 5~6 个任务：

pool.submit_task(相机线程);        // 线程0 干：不停从相机采图
pool.submit_task(USB相机线程);     // 线程1 干：从USB相机采图（如果有）
pool.submit_task(串口线程);        // 线程2 干：和硬件通信
pool.submit_task(装甲板处理线程);   // 线程3 干：识别装甲板、做算法
pool.submit_task(ROS2通信线程);    // 线程4 干：和ROS2系统通信
                                   // 线程5 闲着呢，等新任务
```

```
                            BS::thread_pool (6个线程)
                    ┌──────────────────────────────────────┐
                    │                                      │
  submit_task ──→  │  线程0: 不停采图（HikCamera）          │
  submit_task ──→  │  线程1: 不停采图（USBCamera）          │
  submit_task ──→  │  线程2: 串口收发                      │
  submit_task ──→  │  线程3: 装甲板算法（吃图→识别→输出）   │
  submit_task ──→  │  线程4: ROS2通信                      │
                    │  线程5: 空闲                         │
                    └──────────────────────────────────────┘
```

## 五、BS::thread_pool 内部是怎么工作的？

### 核心数据结构

线程池内部有三个关键零件：

```cpp
// BS_thread_pool.hpp 的私有成员（简化后）
std::queue<task_t> tasks;           // ① 任务队列（"排队等干的活"）
std::mutex         tasks_mutex;     // ② 互斥锁（"同一时间只能一个人碰队列"）
std::condition_variable task_available_cv;  // ③ 条件变量（"通知：有活来了！"）
```

### 零件 ①：任务队列 `tasks`

就是一个 **FIFO 队列**（先进先出），像排队买奶茶：

```
队尾（新任务加到这儿）                  队首（线程从这儿取活）
     │                                        │
     ▼                                        ▼
  [任务5] ← [任务4] ← [任务3] ← [任务2] ← [任务1]
                                        ↑
                                  线程取走这个去干活
```

### 零件 ②：互斥锁 `tasks_mutex`

**问题：** 6 个线程同时去队列里抢活，如果两个线程同时伸手拿同一个任务怎么办？

**解决：** 互斥锁 = 队列门口的一把锁。谁拿到锁谁才能进去操作队列，其他人门口等着。

```cpp
// 每次操作队列前：
{
    const std::scoped_lock tasks_lock(tasks_mutex);  // 上锁
    // ... 安全地操作队列 ...
}                                                    // 自动解锁
```

`std::scoped_lock` 就是 RAII 的锁——创建时自动上锁，离开 `{}` 时自动解锁。哪怕中间代码崩了，锁也一定解开。

### 零件 ③：条件变量 `task_available_cv`

**问题：** 队列空了，线程怎么办？不能一直死循环空等（浪费CPU）。

**解决：** 条件变量 = 一个"呼叫铃"。

```
线程: "队列空了，我去睡觉了" → task_available_cv.wait(...)  // 线程阻塞，不占CPU
                                    ↓
submit_task 往队列加了一个任务
            ↓
            按铃: task_available_cv.notify_one()   // "醒醒，有活了！"
            ↓
睡着的线程被唤醒: "来了来了！" → 从队列取活干
```

## 六、worker() 函数 —— 线程的"一生"

每个线程启动后，就跑 `worker()` 函数。这个函数是一个**死循环**，线程在循环里过完一生。

### worker 函数简化版

```cpp
void worker(线程编号 idx)
{
    // ① 报到：记录"我是谁，我属于哪个池"
    this_thread::my_pool = this;      // 我是这个池的人
    this_thread::my_index = idx;      // 我是第 idx 号线程

    // ② 初始化：干点准备工作
    init_func(idx);

    // ③ 进入循环：一辈子都在接活→干活→接活→干活...
    while (true)
    {
        // Step 1: 告诉池子"我刚干完一个活，现在闲了"
        --tasks_running;

        // Step 2: 检查队列，队列空的 → 去睡觉（等通知）
        task_available_cv.wait(...);
        //      ↑ 线程在这里挂起，不占CPU。直到有人按铃才醒。

        // Step 3: 如果收到"停工令" → 结束循环，退休
        if (收到停止信号)
            break;

        // Step 4: 从队列取一个任务
        task_t task = pop_task();
        ++tasks_running;    // "我开始干活了"
        解锁;

        // Step 5: 执行任务
        task();             // ← 真正干活！
    }

    // ④ 退休前清理
    cleanup_func(idx);
    this_thread::my_index = std::nullopt;  // "我不再属于任何池了"
    this_thread::my_pool = std::nullopt;
}
```

### 一张图看懂线程的一生

```
创建 → 报到 → 初始化 → 睡等 → 拿活 → 干活 → 睡等 → 拿活 → ...
                        ↑                        │
                        └──── 循环 ──────────────┘
                                                     ↓
                                                  退休 → 消失
```

关键点：**线程只在"干活"时才消耗CPU，睡觉时完全不占资源。**

## 七、两种"交活"方式对比

BS::thread_pool 提供了两大类接口：

| 接口名 | 有没有"收据" | 能拿返回值吗 |
|--------|------------|------------|
| `submit_task()` | ✅ 有 future | ✅ 能 |
| `submit_blocks()` | ✅ 有 multi_future | ✅ 能 |
| `detach_task()` | ❌ 无 | ❌ 不能 |
| `detach_blocks()` | ❌ 无 | ❌ 不能 |

### submit（有收据）

```cpp
auto future = pool.submit_task([]() { return 42; });
// ... 干别的事 ...
int result = future.get();   // 拿到 42
```

### detach（没有收据）

```cpp
pool.detach_task([]() { /* 干活，不返回任何东西 */ });
// 扔进池子就不管了，没法知道什么时候干完
```

## 九、main.cpp 中的线程间通信

线程之间怎么传数据？用 **SPSCQueue**（单生产者单消费者队列）。

```
相机线程                          装甲板线程
（生产者）                        （消费者）
    │                                │
    │  不停拍照片                    │  不停处理照片
    │                                │
    └──── SPSCQueue ────────────────┘
         （管道）
```

```cpp
// main.cpp 第 82-83 行
auto camera_buffer = std::make_shared<rigtorp::SPSCQueue<CameraFrame>>(2);

// 第 86-87 行：相机线程往 buffer 里塞图
pool.submit_task([camera_buffer]() {
    threads::camera_thread_func(..., camera_buffer);
});

// 第 103 行：装甲板线程从 buffer 里取图
pool.submit_task([camera_buffer]() {
    threads::armor_thread_func(camera_buffer, ...);
});
```

一张图理解：

```
相机线程：                    装甲板线程：
"咔嚓"                         "这张图里有装甲板吗？"
  │                              ↑
  │  塞进队列                    │  从队列取出
  ▼   ┌─────────────────┐       │
  ├──→│ SPSCQueue(容量2) │──→────┘
      └─────────────────┘
      最多存2张图，超过就丢旧的
```

## 十、程序怎么优雅退出？

```cpp
// main.cpp 第 24 行
std::atomic<bool> g_shutdown{false};  // 全局"停工旗"

// 第 122-125 行：主线程等信号
while (!g_shutdown)
{
    std::this_thread::sleep_for(std::chrono::milliseconds(100));
}
// 主线程每 0.1 秒抬头看一眼"是不是该下班了？"

// 每个工作线程里（以相机线程为例）：
while (!g_shutdown)     // 也抬头看一眼
{
    // 干活...
}
```

当用户按 Ctrl+C：

```
Ctrl+C
  ↓
signal_handler() 被调用
  ↓
g_shutdown = true         ← 升起"停工旗"
  ↓
所有线程看到旗子          ← "哦，该下班了"
  ↓
各自退出循环，关闭设备
  ↓
主线程等所有人退出        ← camera_future.get() ... 等5个线程都走
  ↓
程序结束 ✅
```

## 十一、关键概念速查表

| 语法 / 概念 | 简单解释 | 类比 |
|------------|---------|------|
| **线程** (`std::thread`) | 一个独立运行的执行流 | 一个厨师 |
| **线程池** | 固定数量的线程 + 任务队列 | 提前招好的厨师团队 + 排队等干的活 |
| **`submit_task()`** | 把活交给线程池，拿回一个收据 | 把衣服给干洗店，拿回一张取衣单 |
| **`future`** | 收据，用来等结果/拿结果 | 取衣单 |
| **`future.get()`** | 凭收据拿结果（阻塞直到完成） | 凭取衣单取衣服 |
| **`mutex`（互斥锁）** | 同一时间只让一个线程访问共享数据 | 厕所门锁，一次只能进一个人 |
| **`condition_variable`（条件变量）** | 线程睡觉等待，直到被通知 | 呼叫铃，"有活了按铃叫醒我" |
| **`lambda` 表达式 `[](){}`** | 匿名函数，打包一段代码 | 把要做的事写在一张便签上 |
| **`atomic<bool>`** | 多线程安全的布尔变量 | 所有人能同时看、不会读错的公告牌 |
| **SPSCQueue** | 单生产者单消费者队列 | 传送带：一头放东西，一头取东西 |
| **异步** | 主线程不等待，丢给后台干 | 点外卖：点完干别的，外卖到了再说 |
| **同步** | 主线程等着，直到完成 | 自己做饭：必须站着等饭熟 |

## 十二、总结

```
你的程序启动后，有 7 个线程在同时运行：

    主线程（main）：   创建线程池 → 分配任务 → 睡觉 → 等Ctrl+C → 收尾
    spdlog 后台线程：  不停写日志，安安静静，不打扰任何人
    BS::线程0：        相机不停地采图
    BS::线程1：        USB相机不停地采图
    BS::线程2：        串口不停地收发
    BS::线程3：        不停地处理装甲板
    BS::线程4：        ROS2 不停地通信

它们就像一栋写字楼里的 7 个人，各干各的，通过 SPSCQueue 传递数据，
通过 g_shutdown 信号统一下班，通过互斥锁避免抢资源。
```
